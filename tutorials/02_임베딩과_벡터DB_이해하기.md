# 02. 임베딩과 벡터DB 이해하기

## 이 문서의 목표
> 임베딩이 뭔지, 왜 필요한지, 벡터DB가 어떻게 "의미 기반 검색"을 가능하게 하는지 이해합니다.

---

## 1. 임베딩(Embedding)이란?

### 한 줄 요약
> 텍스트를 **숫자 배열(벡터)**로 변환하는 것. 의미가 비슷한 문장은 비슷한 숫자가 됩니다.

### 왜 숫자로 바꿔야 하나?

컴퓨터는 텍스트의 "의미"를 직접 이해하지 못합니다.
하지만 숫자라면 **비교**할 수 있습니다.

```
"고양이는 귀엽다"  →  [0.82, 0.15, 0.43, 0.91, ...]   ─┐
                                                          ├── 숫자가 비슷! = 의미가 비슷!
"냥이는 사랑스럽다" →  [0.80, 0.17, 0.45, 0.89, ...]   ─┘

"오늘 날씨가 좋다"  →  [0.12, 0.67, 0.33, 0.05, ...]   ← 숫자가 다름 = 의미가 다름
```

### 임베딩의 핵심 특성

```
1. 고차원 벡터
   - 텍스트 하나가 수백~수천 개의 숫자로 변환됨
   - OpenAI text-embedding-3-small: 1536차원
   - 차원이 높을수록 더 세밀한 의미 표현 가능

2. 의미적 유사성 보존
   - "사과"와 "배"의 벡터는 가까움 (둘 다 과일)
   - "사과"와 "자동차"의 벡터는 멂 (관계없음)

3. 방향성
   - 왕 - 남자 + 여자 ≈ 여왕  (유명한 예시!)
   - 벡터 연산으로 의미 관계를 표현할 수 있음
```

### 비유: 도서관의 분류 번호

```
기존 키워드 검색 = 책 제목에 "고양이"라는 글자가 있는 책 찾기
  → "냥이", "cat", "반려묘" 는 검색 안 됨 ❌

임베딩 검색 = 고양이에 "관한" 책 찾기
  → "냥이", "cat", "반려묘" 모두 검색됨 ✅
```

## 2. 코사인 유사도 (Cosine Similarity)

두 벡터가 얼마나 비슷한지 측정하는 방법입니다.

```
코사인 유사도 = 두 벡터가 이루는 각도의 코사인 값

  1.0  = 완전히 같은 방향 (= 의미가 거의 같음)
  0.0  = 직각 (= 관계 없음)
 -1.0  = 반대 방향 (= 의미가 반대)
```

### 시각적 이해

```
       ↑ 차원2
       │
       │    ★ "고양이는 귀엽다"
       │   /
       │  / ← 각도가 작음 = 유사도 높음
       │ /
       │/ ★ "냥이는 사랑스럽다"
       ├─────────────────→ 차원1
       │
       │
       │          ★ "오늘 비가 온다" (각도가 큼 = 유사도 낮음)
```

### Python으로 확인해보기

```python
from langchain_openai import OpenAIEmbeddings
import numpy as np

embeddings = OpenAIEmbeddings(model="text-embedding-3-small")

# 3개 문장의 임베딩 생성
texts = ["고양이는 귀엽다", "냥이는 사랑스럽다", "오늘 날씨가 좋다"]
vectors = embeddings.embed_documents(texts)

# 코사인 유사도 계산
def cosine_similarity(a, b):
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

print(f"고양이 vs 냥이:  {cosine_similarity(vectors[0], vectors[1]):.4f}")  # 높음!
print(f"고양이 vs 날씨:  {cosine_similarity(vectors[0], vectors[2]):.4f}")  # 낮음!
```

## 3. 텍스트 분할 (Text Splitting)

### 왜 분할하나?

```
문제 1: 문서가 너무 길면 임베딩 품질이 떨어짐
  - 10페이지짜리 문서 전체를 하나의 벡터로? → 의미가 희석됨
  - "연차 정책"과 "급여 정책"이 섞여서 검색 정확도 ↓

문제 2: LLM 컨텍스트 윈도우 제한
  - 검색된 문서를 LLM에 넣어야 하는데, 너무 길면 안 들어감

해결: 적절한 크기로 잘라서(chunk) 각각 임베딩
```

### 분할 전략

```
┌──────────────────────────────────────────┐
│           원본 문서 (1000자)              │
│                                          │
│  1. 근무시간 정책                         │
│  - 9시~6시 근무                           │
│  - 유연근무 가능                          │
│                                          │
│  2. 휴가 정책                            │
│  - 연차 15일                             │
│  - 경조사 휴가                            │
│                                          │
│  3. 복리후생                             │
│  - 식대 지원                             │
│  - 건강검진                              │
└──────────────────────────────────────────┘
        │
        ▼  RecursiveCharacterTextSplitter(chunk_size=300, chunk_overlap=50)
        │
┌──────────────┐  ┌──────────────┐  ┌──────────────┐
│   청크 1      │  │   청크 2      │  │   청크 3      │
│  근무시간 정책 │  │  휴가 정책    │  │  복리후생      │
│  (300자)      │  │  (300자)      │  │  (250자)      │
│         ░░░░░│──│░░░░░         │  │         ░░░░░│
│         겹침  │  │ 겹침          │  │               │
└──────────────┘  └──────────────┘  └──────────────┘
                      ↑
               chunk_overlap=50
            (앞뒤 문맥이 끊기지 않게)
```

### 분할 파라미터 가이드

| 파라미터 | 작게 설정 | 크게 설정 |
|---------|-----------|-----------|
| **chunk_size** | 정밀한 검색, 문맥 부족 가능 | 풍부한 문맥, 노이즈 증가 가능 |
| **chunk_overlap** | 문맥 끊김 위험 | 중복 증가, 저장 공간 ↑ |

```
추천 시작값:
  chunk_size = 300~500  (한국어는 글자 수가 적으므로 300 정도가 적당)
  chunk_overlap = chunk_size의 10~20% (30~100)
```

## 4. 벡터 DB (Vector Database)

### 벡터 DB란?

> 벡터(숫자 배열)를 저장하고, **"가장 비슷한 벡터"를 빠르게 검색**할 수 있는 데이터베이스

### 일반 DB vs 벡터 DB

```
일반 DB (MySQL 등):
  SELECT * FROM docs WHERE title = '연차'    ← 정확히 일치하는 것만 찾음
  → "휴가", "연차휴가", "annual leave" 못 찾음

벡터 DB (Chroma 등):
  query_vector = embed("연차가 궁금해요")
  search(query_vector, top_k=3)              ← 의미가 비슷한 것을 찾음
  → "연차는 15일", "휴가 정책", "annual leave" 모두 찾음!
```

### 주요 벡터 DB 비교

| 벡터 DB | 특징 | 적합한 경우 |
|---------|------|------------|
| **Chroma** | 설치 쉬움, 로컬 파일 저장 | 학습, 프로토타입, 소규모 |
| **FAISS** | Meta 제작, 매우 빠름 | 대규모 검색, 로컬 |
| **Pinecone** | 클라우드 서비스, 관리형 | 프로덕션, 서버리스 |
| **Weaviate** | 오픈소스, 풍부한 기능 | 중대규모 프로덕션 |
| **Qdrant** | Rust 기반, 고성능 | 고성능이 필요한 프로덕션 |

> **초보자 추천**: Chroma로 시작 → 익숙해지면 FAISS나 Pinecone으로 전환

### 벡터 DB의 동작 원리

```
[저장 과정]
  "연차는 15일" → embed → [0.82, 0.15, ...] → ChromaDB에 저장
  "식대 20만원"  → embed → [0.23, 0.67, ...] → ChromaDB에 저장
  "재택 주2회"   → embed → [0.45, 0.33, ...] → ChromaDB에 저장

[검색 과정]
  "휴가 정책?"  → embed → [0.80, 0.17, ...] → 가장 비슷한 벡터 검색
                                                ↓
                                          [0.82, 0.15, ...] "연차는 15일" ← 1등!
                                          [0.45, 0.33, ...] "재택 주2회"  ← 2등
                                          [0.23, 0.67, ...] "식대 20만원" ← 3등
```

## 5. 검색 전략

### Similarity Search (기본)

```
가장 가까운 k개 벡터를 반환
장점: 단순하고 빠름
단점: 비슷한 내용끼리 중복될 수 있음

예: k=3으로 "연차" 검색 →
  1. "연차는 15일입니다"
  2. "연차는 1년 이상 근무 시 부여됩니다"    ← 비슷한 내용 중복
  3. "연차 소진 시 무급휴가 가능합니다"       ← 비슷한 내용 중복
```

### MMR (Maximal Marginal Relevance)

```
관련성 높으면서도 서로 다양한 내용의 문서를 반환
장점: 중복 없이 다양한 정보 제공
단점: 약간 느림

예: k=3으로 "연차" 검색 →
  1. "연차는 15일입니다"                     ← 가장 관련 높음
  2. "재택근무는 주 2회 가능"                 ← 다른 근무 정책
  3. "경조사 휴가: 결혼 5일"                  ← 다른 종류의 휴가
```

## 핵심 정리

```
┌─────────────────────────────────────────────────────┐
│                                                     │
│  텍스트 → [임베딩] → 숫자 벡터 → [벡터DB 저장]       │
│                                                     │
│  질문 → [임베딩] → 숫자 벡터 → [벡터DB 검색]         │
│                                    ↓                │
│                              유사한 문서 반환         │
│                                                     │
│  기억할 것:                                          │
│  • 임베딩 = 의미를 숫자로                             │
│  • 벡터DB = 의미로 검색하는 DB                        │
│  • 텍스트 분할 = 적절한 크기가 검색 품질의 핵심        │
│                                                     │
└─────────────────────────────────────────────────────┘
```
