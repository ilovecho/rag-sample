## 실수 1: chunk_size가 너무 크거나 작음

### 증상
```
- 검색 결과가 엉뚱한 내용을 반환한다
- 답변이 너무 일반적이거나, 중요한 세부 정보가 누락된다
```

### 원인과 해결
```
chunk_size가 너무 큰 경우 (예: 2000자):
  ❌ 하나의 청크에 여러 주제가 섞임
  ❌ "연차" 검색 시 연차+급여+복지가 한 덩어리로 나옴
  → 검색 정밀도가 떨어짐

chunk_size가 너무 작은 경우 (예: 50자):
  ❌ 문장이 잘려서 의미가 불완전
  ❌ "연차는 15일" → "연차는" / "15일" 로 분리될 수 있음
  → 맥락이 끊김

✅ 해결: 300~500자로 시작, 결과를 보면서 조정
✅ chunk_overlap을 chunk_size의 10~20%로 설정
```

### 확인하는 방법
```python
# 분할 결과를 직접 눈으로 확인!
for i, chunk in enumerate(chunks):
    print(f"\n[청크 {i}] ({len(chunk.page_content)}자)")
    print(chunk.page_content)
    print("---")
# → 각 청크가 하나의 완결된 의미를 담고 있는지 확인
```

## 실수 2: 검색된 문서가 엉뚱함

### 증상
```
- "연차 정책"을 물었는데 "제품 요금제"가 검색됨
- 관련 없는 문서가 상위에 올라옴
```

### 원인과 해결

```
원인 1: k 값이 너무 큼
  k=10 → 관련 없는 문서까지 포함됨
  ✅ k=2~4로 줄여보세요

원인 2: 문서 종류가 섞여 있음
  회사 정책 + 제품 매뉴얼이 같은 벡터DB에 저장
  ✅ 메타데이터로 카테고리 분류 후 필터링 검색

원인 3: 임베딩 모델의 한계
  저렴한 모델은 한국어 의미 파악이 약할 수 있음
  ✅ text-embedding-3-large로 업그레이드 시도
```

### 디버깅 방법
```python
# 검색 결과를 유사도 점수와 함께 확인
results = vectorstore.similarity_search_with_score(query, k=5)
for doc, score in results:
    print(f"  점수: {score:.4f} | {doc.page_content[:60]}...")

# → 점수가 갑자기 크게 떨어지는 지점이 있으면
#   그 이하는 관련 없는 문서일 가능성이 높음
```

## 실수 3: API 키 에러

### 증상
```
openai.AuthenticationError: Incorrect API key provided
```

### 해결
```
1. .env 파일이 있는지 확인
   ls -la .env

2. .env 파일 내용 확인 (따옴표 없이!)
   ✅ OPENAI_API_KEY=sk-abc123...
   ❌ OPENAI_API_KEY="sk-abc123..."    ← 따옴표 없어야 함
   ❌ OPENAI_API_KEY = sk-abc123...    ← 공백 없어야 함

3. 코드에서 load_dotenv() 호출 확인
   from dotenv import load_dotenv
   load_dotenv()  # 이게 빠지면 .env를 못 읽음!

4. API 키 유효성 확인
   https://platform.openai.com/api-keys 에서 확인
```

## 실수 4: 한국어 인코딩 에러

### 증상
```
UnicodeDecodeError: 'utf-8' codec can't decode byte...
또는
UnicodeDecodeError: 'cp949' codec can't decode byte...
```

### 해결
```python
# TextLoader에 encoding 지정
loader = TextLoader("파일.txt", encoding="utf-8")

# 파일이 cp949(Windows 한국어)로 저장된 경우
loader = TextLoader("파일.txt", encoding="cp949")

# 인코딩을 모르겠다면
loader = TextLoader("파일.txt", encoding="utf-8-sig")  # BOM 포함 UTF-8
```

## 실수 5: 답변이 문서 내용을 무시함

### 증상
```
- 문서에 "연차 15일"이라고 써있는데 LLM이 "일반적으로 연차는..."이라고 답변
- 문서 근거 없이 일반적인 지식으로 답변
```

### 해결
```
원인: 프롬프트가 문서 참조를 강제하지 않음

✅ 프롬프트에 명시적으로 지시:
```
```python
prompt = PromptTemplate(
    template="""
반드시 아래 [참고 문서]만을 근거로 답변하세요.    ← 핵심!
참고 문서에 없는 내용은 "확인되지 않습니다"라고 답하세요.  ← 핵심!

[참고 문서]
{context}

[질문] {question}
[답변]""",
    input_variables=["context", "question"],
)
```

## 실수 6: 메모리(RAM) 부족

### 증상
```
- 프로그램이 느려지거나 멈춤
- MemoryError 발생
```

### 해결
```
원인 1: 너무 많은 문서를 한 번에 로드
  ✅ 문서를 배치(batch)로 나눠서 처리

원인 2: ChromaDB 데이터가 너무 커짐
  ✅ persist_directory 지정해서 디스크 저장 활용
  ✅ 대규모 데이터는 FAISS나 Pinecone 고려

원인 3: 임베딩을 반복 생성
  ✅ persist_directory로 저장하면 재시작 시 다시 안 만들어도 됨
```

```python
# 저장된 벡터DB 재사용 (임베딩 비용 절약!)
from langchain_community.vectorstores import Chroma
from langchain_openai import OpenAIEmbeddings

# 처음 한 번만 생성
vectorstore = Chroma.from_documents(
    chunks, embeddings, persist_directory="./chroma_db"
)

# 이후 재사용 (임베딩 다시 안 함!)
vectorstore = Chroma(
    persist_directory="./chroma_db",
    embedding_function=OpenAIEmbeddings(model="text-embedding-3-small"),
)
```

## 실수 7: PDF 텍스트 추출 실패

### 증상
```
- PDF를 로드했는데 빈 텍스트가 나옴
- 텍스트가 깨져서 나옴
```

### 해결
```
원인 1: 스캔된 이미지 PDF (텍스트가 아닌 이미지)
  ✅ OCR 라이브러리 사용 (pytesseract, unstructured)

원인 2: 한글 PDF 호환성
  ✅ PyPDFLoader 대신 다른 로더 시도

원인 3: 암호화된 PDF
  ✅ 비밀번호 해제 후 시도
```

```python
# PDF 로더 대안들
from langchain_community.document_loaders import PyPDFLoader      # 기본
from langchain_community.document_loaders import PDFMinerLoader    # 더 정확
from langchain_community.document_loaders import UnstructuredPDFLoader  # 가장 강력

# 텍스트 추출 확인
loader = PyPDFLoader("document.pdf")
pages = loader.load()
for page in pages:
    print(f"p.{page.metadata['page']}: {len(page.page_content)}자")
    if len(page.page_content) < 10:
        print("  ⚠️ 텍스트가 거의 없음! OCR이 필요할 수 있습니다.")
```

## 실수 8: 비용이 너무 많이 나옴

### 비용 발생 지점
```
1. 임베딩 API 호출 — 문서를 벡터로 변환할 때
2. LLM API 호출   — 답변을 생성할 때
```

### 비용 절약 팁
```
✅ 임베딩: text-embedding-3-small 사용 (large 대비 5배 저렴)
✅ LLM: gpt-4o-mini 사용 (gpt-4o 대비 30배 저렴)
✅ 벡터DB를 persist해서 임베딩 재생성 방지
✅ 개발 중에는 k=2로 검색 개수 줄이기
✅ 불필요한 문서는 벡터DB에서 제외

비용 참고 (2024년 기준, 변동 가능):
  text-embedding-3-small: $0.02 / 1M 토큰
  gpt-4o-mini:           $0.15 / 1M 입력 토큰
  → 일반적인 RAG 프로토타입은 월 $1~5 수준
```
